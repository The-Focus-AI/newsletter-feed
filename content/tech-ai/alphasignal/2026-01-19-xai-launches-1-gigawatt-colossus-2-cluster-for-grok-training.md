Cursor outlines how Bugbot scaled to millions of pull requests with agentic reviews
1,037 Likes
Bugbot began as an internal project at Cursor after coding agents reduced writing time but increased review load. Pull requests became larger, review time grew, and logic bugs escaped more often.
The team responded by building Bugbot and defining success as one concrete signal: whether engineers actually fix the bugs it reports.
Initial versions used fixed pipelines with multiple parallel model passes to control false positives. That approach improved signal quality but limited further gains.
The key change came when Cursor rebuilt Bugbot as a fully agentic system. Bugbot now inspects diffs, decides what code paths look risky, pulls additional repository context on demand, and uses tools to investigate issues instead of following a preset sequence.
To evaluate progress, the team introduced
resolution rate
, an AI-judged metric measured at pull request merge time that checks whether reported issues were resolved in final code.
What Bugbot does
Reviews pull requests for logic bugs, performance regressions, and security issues
Filters false positives using validator models and majority signals
Applies repository-specific rules for internal APIs and unsafe patterns
Results and usage
Resolution rate increased from
52%
to
over 70%
Resolved bugs per pull request increased from
0.2
to
0.5
Reviews more than
2 million pull requests per month
You use it by connecting a GitHub repository and running it on pull requests
